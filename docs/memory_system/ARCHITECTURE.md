## 一、大目标

### 1.1 设计初衷

构建一个**真正可用**的 AI Agent 记忆系统，解决人类记忆的五大局限：

| 人类记忆局限 | Memory 系统目标 |
|-------------|----------------|
| **会遗忘** | 全细节留存，永不丢失 |
| **难关联** | 跨时空关联，智能检索 |
| **易固化** | 动态演化，偏好可变 |
| **弱多模态** | 支持图片、语音等多模态 |
| **检索慢** | 毫秒级检索，即时响应 |

### 1.2 核心能力要求

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Memory 系统核心能力                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 信息提取                                                                │
│     ├── 正则提取：过敏、偏好、决策、联系方式等固定模式                      │
│     └── LLM 提取：项目背景、技术决策、人际关系等复杂语义                    │
│                                                                             │
│  2. 持久存储                                                                │
│     ├── 结构化存储：Entity-Relation 模型                                    │
│     ├── 向量索引：支持语义检索                                              │
│     └── 持久化：JSON 文件，跨 Session 保留                                  │
│                                                                             │
│  3. 智能检索                                                                │
│     ├── 关键词检索：精确匹配                                                │
│     ├── 语义检索：向量相似度                                                │
│     └── 混合检索：结合多种策略                                              │
│                                                                             │
│  4. 动态注入                                                                │
│     ├── 安全优先：过敏、禁忌必须注入                                        │

│     └── Token 预算：控制注入内容长度                                        │

│     ├── 冲突处理：智能处理矛盾偏好                                          │
│     └── 置信度衰减：长时间未提及的偏好自动降权                              │
│                                                                             │
│  6. 事件追踪                                                                │
│     ├── 里程碑记录：重要事件和里程碑                                        │

│     └── 周期事件：重复性事件管理                                            │
```

## 三、里程碑定义

### 3.1 里程碑总览

| 里程碑 | 名称 | 核心能力 | 状态 |
|--------|------|----------|------|
| **M2.1** | 关键信息保护 | 正则提取 + 摘要保护 | ✅ 已完成 |
| **M3.0** | 偏好演化 + 事件追踪 | 偏好变化 + 事件记录 | ✅ 已完成 |
| **M3.5** | 动态检索注入 | 统一存储 + 检索 + 注入 | ⚠️ 开发中，未集成 |
| **M4.0** | LLM 语义提取 | 复杂语义理解 | ⚠️ 开发中，未集成 |

### 3.2 各里程碑详细定义

#### M2.1 关键信息保护

**目标**: 确保重要信息在对话压缩时不丢失

**能力**:
- 正则模式提取：过敏、偏好、决策、联系方式
- 两层保护机制：注入关键信息到压缩 prompt + 验证摘要

**输出**:
- `KeyInfo` 数据结构
- `KeyInfoExtractor` 提取器

**状态**: ✅ 已完成，已集成到 `memory_compaction.py`

---

#### M3.0 偏好演化 + 事件追踪

**目标**: 记录偏好变化历史，追踪重要事件

**能力**:
- 偏好演化：
  - 记录偏好变化时间线
  - 冲突检测和处理
  - 置信度衰减算法
- 事件追踪：
  - 里程碑记录
  - 截止日期提醒
  - 周期事件管理

**输出**:
- `Preference` 数据模型
- `PreferenceManager` 管理器
- `KeyEvent` 数据模型
- `EventTracker` 追踪器
- `MemoryV3` 统一接口

**状态**: ✅ 已完成，独立模块存在

---

#### M3.5 动态检索注入

**目标**: 修复三层断链，实现信息提取→存储→检索→注入的完整链路

**能力**:
- 统一存储：所有提取结果存入 `UnifiedEntityStore`
- 智能检索：关键词检索 + 语义检索
- 动态注入：根据查询动态注入相关实体到 System Prompt

**输出**:
- `Entity` 统一数据模型
- `UnifiedEntityStore` 存储层
- `EntityRetriever` 检索器
- `DynamicInjector` 注入器
- `MemoryIntegration` 集成入口

**状态**: ⚠️ 已开发，未与 M3.0 集成，未接入主系统

---

#### M4.0 LLM 语义提取

**目标**: 突破正则模式限制，实现复杂语义理解

**能力**:
- 项目背景提取
- 技术决策提取
- 人际关系提取
- 条件偏好提取
- 智能触发策略（成本优化）

**输出**:
- `LLMExtractor` 提取引擎
- `LLMExtractedEntity` 数据模型
- `TriggerStrategy` 触发策略
- `V4IntegratedExtractor` 统一提取入口

**状态**: ⚠️ 已开发，未与 M3.5 集成

## 四、当前代码架构问题

### 4.1 问题诊断

```
当前状态（错误）:

src/copaw/agents/memory/
├── preference/          ← M3.0，独立存在，输出未接入 unified
├── events/              ← M3.0，独立存在，输出未接入 unified
├── memory_v3.py         ← M3.0 统一接口，未被使用
├── memory_manager.py    ← M2.1，被 react_agent 使用
└── unified/             ← M3.5/M4.0，独立存在
    ├── models.py        ← Entity 模型
    ├── store.py         ← 未接收 M3.0 的输出
    ├── retriever.py
    ├── injector.py      ← 未接入 react_agent
    ├── integration.py   ← 未被任何模块调用
    ├── llm_extractor.py ← M4.0，输出未接入 store
    └── v4_integration.py
```

### 4.2 问题清单

| 问题 | 描述 | 影响 |
|------|------|------|
| **断链1** | M3.0 的 preference/events 输出未存入 unified store | 偏好和事件无法被检索注入 |
| **断链2** | M3.5 的 integration.py 未被 react_agent 调用 | 整个 unified 模块未生效 |
| **断链3** | M4.0 的 llm_extractor 输出未接入 unified store | LLM 提取结果丢失 |
| **断链4** | 各模块数据模型不统一 | 无法共享数据 |

## 五、整合计划

### 5.1 Phase 1: 代码整合 (预计 1 天)

1. **统一数据模型**
   - 定义统一的 Entity 模型
   - M3.0 的 Preference/Event 转换为 Entity
   - M4.0 的 LLMExtractedEntity 转换为 Entity

2. **集成提取层**
   - 创建统一提取入口
   - 先执行正则提取（M2.1 + M3.0）

3. **集成到 react_agent**
   - 在 message 处理时调用提取
   - 在 prompt 构建时调用注入

## 六、验收标准

### 6.1 功能验收

| 场景 | 输入 | 期望结果 |
|------|------|----------|
| 安全信息 | "我对花生过敏" | 后续对话中始终记住，优先注入 |
| 偏好记录 | "我喜欢蓝色的杯子" | 存储为 Entity，可检索 |
| 偏好变化 | "我现在喜欢绿色的杯子" | 更新偏好，保留历史 |
| 事件记录 | "明天下午3点开会" | 存储为 Event，可提醒 |
| 项目信息 | "我正在做卫星计算项目" | LLM 提取，存储为 Entity |
| 跨对话记忆 | Session 1 说偏好，Session 2 询问 | 能正确回忆 |

### 6.2 架构验收

- [ ] 所有提取器的输出都存入 UnifiedEntityStore
- [ ] UnifiedEntityStore 支持持久化
- [ ] DynamicInjector 正确注入到 System Prompt
- [ ] M3.0 的 Preference/Event 与 Unified Entity 正确转换
- [ ] M4.0 的 LLM 提取结果正确存储
