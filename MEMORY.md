## 从 Co 的经验中学习 (关键踩坑记录)

### 1. 环境隔离认知 ⚠️

**关键教训**：运行环境 ≠ 开发环境

| 环境 | 位置 | 用途 |
|------|------|------|
| **运行环境** | 当前容器 (site-packages) | 我**运行**的地方，只读，不可开发 |
| **开发环境** | ECS-2 的 `co-dev` 容器 | 我**开发**的地方，可 commit/push |

**错误模式**：在运行环境中开发代码，违反环境隔离原则

### 2. 里程碑增量开发原则 ⚠️

**用户明确指出**：里程碑应该是增量开发：
- M2 在 M1 基础上构建
- M3 在 M1+M2 基础上构建
- M4 在 M1+M2+M3 基础上构建

**常见错误**：开发新功能但未与现有系统集成

### 3. 测试覆盖问题 ⚠️

**V4.0 测试的教训**：
- ❌ 没有真实 LLM API 调用测试
- ❌ 没有大规模数据集测试
- ❌ 没有长对话场景测试
- ❌ 使用模拟数据而非真实场景

**应该创建**：
- `test_xxx_large_scenario.py` - 大规模测试
- `test_xxx_e2e.py` - 端到端真实测试

### 4. 配置路径问题 ⚠️

**SECRET_DIR 路径陷阱**：
- 预期：`/root/.copaw/.secret/providers/`
- 实际：`/root/.copaw.secret/providers/`（`WORKING_DIR + ".secret"` 的问题）

**教训**：路径拼接时要注意 `WORKING_DIR` 是否带末尾斜杠

### 5. LLM 流式响应处理 ⚠️

- 返回的 `content` 可能是列表（包含 `thinking` 和 `text` 类型）

## 短期记忆系统反思 (2026-03-22)

### 观察到的问题

sm-co 在处理多任务指令时的行为：
```
输入：[学习任务1, 学习任务2, 学习任务3, 安全任务]
输出：只执行了安全任务，忽略了学习任务
```

### 可能的技术原因

| 问题 | 说明 |
|------|------|
| 任务优先级偏见 | 安全任务被赋予过高优先级，"冲掉"其他任务 |
| 缺乏任务分解 | 没有把复杂指令拆分成独立子任务 |
| 缺乏状态追踪 | 没有"已完成/待处理"的任务清单机制 |
| 注意力衰减 | 距离当前处理位置越远的指令，权重越低 |
| 单线程执行 | 完成一个任务后直接结束，不回溯检查 |

### 改进方向

1. **显式任务列表**：在处理复杂指令时，先提取所有任务并创建清单
2. **状态追踪**：维护任务完成状态，完成一项后检查是否还有待处理
3. **任务优先级平衡**：安全任务重要，但不应完全覆盖其他任务
4. **回溯机制**：任务完成后主动询问或检查是否有遗漏

| 原则 | 内容 |
